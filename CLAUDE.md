## Architecture Overview

### Core Patterns

#### Generic Endpoint System
The codebase uses Go generics for type-safe API endpoints:
```go
type Endpoint[Request any, Response any, Credits response.Credits, Error error]
```

Each API endpoint follows this pattern:
- **Request struct**: Contains API parameters with `schema` tags for URL encoding
- **Response struct**: Matches API JSON response structure
- **Generic endpoint**: Handles HTTP communication, error parsing, and credits tracking
- **Client method**: Simple wrapper that calls the endpoint

#### Request/Response Architecture
- `request/` directory: Contains all API request parameter structures
- `response/` directory: Contains all API response structures
- Schema tags (`schema:"param_name,omitempty"`) enable automatic URL parameter encoding

#### Error Handling
The system has two layers of error handling:
1. **HTTP errors**: BadRequestError, UnauthorizedError, etc.
2. **Domain errors**: SymbolNotFoundError, PlanLimitationError, etc.

Domain errors are automatically parsed from API responses for better UX.

### Endpoint Organization
The codebase follows the exact structure from Twelve Data API documentation. Endpoints are organized in categories:

| Documentation Category | Code Structure | Examples |
|------------------------|-----------------|----------|
| Core Data | `CoreData` struct in conf.go | Time Series, Quote, Price |
| Fundamentals | `Fundamentals` struct in conf.go | Logo, Profile, Dividends, Splits |
| Reference Data | `ReferenceData` struct in conf.go | Stocks, Exchanges, Countries |
| Currencies | `Currencies` struct in conf.go | Exchange Rate |
| Advanced | `Advanced` struct in conf.go | Usage |

**IMPORTANT**: Always place new endpoints in the correct category matching the documentation structure. This maintains code organization and makes it easier to find endpoints.

## Adding New Endpoints

### Step-by-Step Guide

When adding a new endpoint (e.g., `/new_endpoint`), follow these steps:

#### 0. Research and Validate API (REQUIRED FIRST STEP)
1. **Research documentation**: Use the local `doc.html` file to get complete endpoint details:
   ```bash
   # First check local documentation
   Grep(pattern: "endpoint_name", path: "/path/to/twelvedata/doc.html")
   
   # Find credit costs
   Grep(pattern: "API credits cost", path: "/path/to/twelvedata/doc.html") 
   ```
   If local documentation is unavailable, use WebFetch to get details from https://twelvedata.com/docs
2. **Test with demo key**: Make real API calls to understand the actual response structure
   ```bash
   curl -s "https://api.twelvedata.com/new_endpoint?apikey=demo&symbol=AAPL" | jq '.'
   ```
3. **Document any discrepancies**: Note differences between documentation and actual API responses

Then modify these files in order:

#### 1. Create Request Structure (`request/new_endpoint.go`)
```go
package request

type GetNewEndpoint struct {
    ApiKey
    Symbol    string `schema:"symbol,omitempty"`
    Exchange  string `schema:"exchange,omitempty"`
    // Add other parameters based on API documentation
    Range     string `schema:"range,omitempty"`
    StartDate string `schema:"start_date,omitempty"`
    EndDate   string `schema:"end_date,omitempty"`
}
```

**Key patterns:**
- Always embed `ApiKey`
- Use `omitempty` for optional parameters
- Follow naming: `GetEndpointName` struct
- Date parameters: `StartDate`, `EndDate` as strings
- Use exact schema names from API docs

#### 2. Create Response Structure (`response/new_endpoint.go`)
```go
package response

type NewEndpoint struct {
    Meta NewEndpointMeta `json:"meta"`
    Data []DataItem      `json:"data"` // or direct fields
}

type NewEndpointMeta struct {
    Symbol   string `json:"symbol"`
    Exchange string `json:"exchange"`
    // Add other meta fields
}

type DataItem struct {
    Date  string  `json:"date"`
    Value float64 `json:"value"`
    // Add other data fields
}
```

**Key patterns:**
- Match exact JSON field names from API
- Use `null.Float`, `null.Int` for nullable values
- Separate Meta and Data structures when applicable

#### 3. Add URL Configuration (`conf.go`)
**CRITICAL**: Add the URL in the correct section that matches the API documentation category:

```go
// For Fundamentals endpoints
type Fundamentals struct {
    // ... existing fields ...
    NewEndpointURL string `default:"/new_endpoint" json:"new_endpoint_url"`
}

// For Core Data endpoints
type CoreData struct {
    // ... existing fields ...
    NewEndpointURL string `default:"/new_endpoint" json:"new_endpoint_url"`
}
```

**Structure mapping**:
- Fundamentals → `Fundamentals` struct
- Core Data → `CoreData` struct  
- Reference Data → `ReferenceData` struct
- Currencies → `Currencies` struct
- Advanced → `Advanced` struct

#### 4. Add Method to Client Interface (`client.go`)
**IMPORTANT**: Add the method in the correct section with proper category comment:

```go
type Client interface {
    // ... existing methods ...
    
    // Fundamentals (if this is a Fundamentals endpoint)
    GetLogo(request.GetLogo) (response.Logo, response.Credits, error)
    GetProfile(request.GetProfile) (response.Profile, response.Credits, error)
    GetNewEndpoint(request.GetNewEndpoint) (response.NewEndpoint, response.Credits, error)
    
    // Core Data (if this is a Core Data endpoint)
    GetTimeSeries(request.GetTimeSeries) (response.TimeSeries, response.Credits, error)
    GetNewEndpoint(request.GetNewEndpoint) (response.NewEndpoint, response.Credits, error)
}
```

**Follow the existing sectioning pattern** - methods are grouped by category with comments matching the documentation structure.

#### 5. Implement in Client (`implementation.go`)

**Add field to client struct in the correct section**:
```go

    // ... existing fields grouped by category ...
    
    // Fundamentals (if this is a Fundamentals endpoint)
    getLogo       *Endpoint[request.GetLogo, response.Logo, response.Credits, error]
```

**Maintain the comment-based sectioning** that matches the API documentation structure.

Add method implementation:
```go

```

Add initialization in `NewClient` **in the correct section**:

**Use the correct config section** (cfg.Fundamentals, cfg.CoreData, etc.) that matches where you added the URL.

## Important Technical Details

### Dependencies
- **fasthttp**: HTTP client (not standard net/http)
- **gorilla/schema**: URL parameter encoding
- **guregu/null.v4**: Nullable types for API responses
- **zerolog**: Structured logging

### Error Message Patterns
Domain errors are parsed from these API message patterns:
- Symbol not found: `**SYMBOL** not found: ...`
- Plan limitations: `is not available with your plan` or `is available exclusively with ...`
- API key issues: `invalid api key` or `api key is required`

### Testing Environment
Tests require specific environment variables:
```bash
ROOT_DIR=${PWD}
```
These are automatically set by `make test` but required for manual test runs.

### Credits System
Every API call returns credits information in headers:
- `Api-credits-left`: Remaining credits
- `Api-credits-used`: Credits consumed by this request

This is automatically parsed by the `Endpoint` system.
